Building a 24-Agent AI Workforce: Architecture Decisions That Actually Matter

By Bit Rage Labour • January 15, 2026 • 12 min read

Most "AI agent" projects die in the proof-of-concept phase. They work in demos, impress in meetings, and crumble the moment real users send real data. We built 24 production agents that process thousands of tasks and deliver QA-verified outputs in under 14 seconds. Here's how — and more importantly, here's what we'd do differently.

The Problem: AI Agents That Don't Actually Work

When we started Bit Rage Labour, the AI agent landscape was dominated by two extremes: enterprise platforms that cost six figures and take months to deploy, and open-source frameworks that produce impressive demos but can't handle production load.

We needed something in between. A system that could:

  • Process 20+ different task types (sales outreach, support tickets, content, data extraction, and more)
  • Handle concurrent requests without stepping on each other
  • Guarantee quality on every single output — not 60% of the time, every time
  • Run autonomously 24/7 without babysitting
  • Cost less than $0.10 per task in LLM spend

So we built it from scratch. No LangChain. No AutoGPT. No framework tax. Just Python, FastAPI, and four LLM providers wired together with purpose-built orchestration.

Architecture Decision #1: One Agent Per Task Type

The first decision — and the most important one — was rejecting the "general-purpose agent" pattern entirely.

Most agent frameworks try to build one mega-agent that can handle anything. Give it a system prompt, some tools, and let it figure out the rest. This approach fails in production for a simple reason: the more general an agent is, the worse it performs at every specific task.

Instead, we built 24 specialized agents. Each one has:

  • A single responsibility — the Sales Ops agent writes outreach emails. That's it. It doesn't also do data entry.
  • A custom system prompt optimized for its exact output format
  • Task-specific input validation — the resume writer expects career data, the web scraper expects a URL
  • Dedicated QA criteria — what counts as "good" for a support ticket response is completely different from what counts as "good" for a press release
Task Request → Router → Specialized Agent → QA Gate → Delivery ↓ ↓ Rate Limiter LLM Provider Budget Check (GPT-4o/Claude/ Daily Caps Gemini/Grok)

The router (dispatcher/router.py) maps each incoming task to its agent, checks daily limits and token budgets, then hands off execution. There's no AI deciding which agent to use — it's a deterministic routing table. Fast, predictable, debuggable.

Architecture Decision #2: QA as a Pipeline Stage, Not an Afterthought

This is the single most impactful decision we made. Every output from every agent passes through an automated QA verification step before delivery. Period.

Without QA gates, our agents produced garbage 20-40% of the time. With QA gates, that dropped to under 5%. One architectural decision. 4x quality improvement.

The QA agent checks different criteria per task type:

  • Sales outreach: Does the email reference a real signal? Is the subject line under 60 chars? Is there a clear CTA? Does it avoid spam trigger words?
  • Support tickets: Does the response address the actual issue? Is the severity score reasonable? Is the tone professional?
  • Content repurposing: Is each platform format within character limits? Does LinkedIn use paragraphs, not bullet lists? Is the Twitter version actually tweetable?
  • Document extraction: Are all required fields present? Do the confidence scores make sense? Are dates in valid formats?

If QA fails, the agent retries — up to 3 times with adjusted prompts. If it still fails after 3 attempts, the task is flagged for human review rather than delivering garbage.

Architecture Decision #3: Multi-LLM by Default

We never intended to use four LLM providers. We started with OpenAI. Then OpenAI went down for 3 hours on a Tuesday afternoon while we had active client tasks in the queue.

That was the day we added Anthropic Claude as a fallback. Then Google Gemini. Then xAI Grok. Now our utils/llm_client.py handles provider selection automatically:

# Simplified provider chain
providers = ["openai", "anthropic", "gemini", "grok"]

for provider in providers:
    try:
        result = call_llm(provider, system_prompt, user_message)
        if result:
            return result
    except (RateLimitError, APIError, Timeout):
        continue  # Try next provider

raise AllProvidersFailedError("No LLM provider available")

Each provider has its own token budget and daily limit. If OpenAI hits its rate limit, Claude takes over. If Claude is down, Gemini picks up. The user never knows. The output quality stays consistent because our prompts are provider-agnostic — we don't rely on provider-specific features.

Architecture Decision #4: SQLite Task Queue

We considered Redis. We considered RabbitMQ. We considered Celery. We chose SQLite.

Why? Because our deployment runs on a single Railway container, and the overhead of managing a separate queue service wasn't worth the complexity. SQLite in WAL mode handles concurrent reads and writes just fine for our volume.

# dispatcher/queue.py — WAL mode for concurrent access
conn = sqlite3.connect("data/task_queue.db")
conn.execute("PRAGMA journal_mode=WAL")
conn.execute("PRAGMA busy_timeout=5000")

The task queue stores every request with its status, timestamps, inputs, outputs, and billing info. It's also our audit trail — we can trace any task from intake to delivery to payment.

Lesson Learned

Don't reach for distributed systems when a single-process solution works. We process hundreds of tasks per day on SQLite. When we need to scale beyond a single container, we'll move to PostgreSQL — but not before then. Premature infrastructure is a tax you pay every day.

Architecture Decision #5: FastAPI with Sync Processing

Our API (api/intake.py) accepts task requests via POST and can process them in two modes:

  • Async mode (default): Queues the task, returns a task ID immediately. The client polls for results or receives a webhook callback.
  • Sync mode: Processes the task inline and returns the result in the same HTTP response. Average response time: 8-15 seconds depending on task type.

We started async-only but quickly learned that marketplace integrations (Fiverr, Freelancer) and small clients don't want to deal with webhooks. They want to POST a request and GET a result. Sync mode solved that.

What We'd Do Differently

If we started over today:

  1. Start with 4 agents, not 24. Our first 4 (sales, support, content, docs) generate 90% of revenue. The other 20 were built speculatively. Ship the minimum, validate demand, then expand.
  2. Build the QA pipeline first. We added QA after the agents were built, and retrofitting it was painful. If we'd designed it as a pipeline stage from day one, every agent would have been built with QA in mind.
  3. Use structured outputs from the start. We spent weeks debugging JSON parsing issues because our early prompts returned free-text. Now every agent returns structured data, and the code is 10x cleaner.
  4. Track token costs per task from day one. We flew blind on costs for the first month. Now every task logs its token usage and cost, which lets us price accurately and optimize prompts.
